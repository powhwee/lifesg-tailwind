## ⚠️ Important: Tailwind 4 + Turbopack cache gotcha

When you add a new `--color-{name}` to the `@theme inline` block in `globals.css`, the utility class **may not generate** even after the dev server's HMR reports a CSS rebuild. Symptom: HTML uses the new class (e.g. `text-link-list-link`), but the served CSS bundle doesn't contain a matching rule. Computed colour falls back to `inherit`.

**Why this happens**: Turbopack caches its Tailwind content-scan separately from the CSS rebuild. `touch`-ing the consumer `.tsx` doesn't invalidate the scan. The cache only resets on a **real content edit to globals.css** (or, presumably, a server restart).

**Workaround**: After editing the `@theme inline` block, make one more small edit anywhere in `globals.css` (e.g. add and remove a comment). Then verify with:

```bash
curl -s "$(curl -s http://localhost:3000/<page> | grep -oE 'href="[^"]*\.css[^"]*"' | head -1 | cut -d'"' -f2 | sed 's#&quot;#"#g; s#%5B#[#g; s#%5D#]#g' | xargs -I{} echo "http://localhost:3000{}")" | grep "text-<your-new-utility>"
```

If the rule isn't there, the scan is still stale. Restart the dev server (`kill <next-server pid>` then `npm run dev`).

**This cost me ~30 min of confusion on LinkList**. The component looked correct in source but rendered with the wrong color. Save yourself the debug — verify the served CSS bundle whenever utilities don't seem to apply.
